# Review 2026-04-10 — Phase 2.5 Playbook

**审查对象**：
- `playbook/phase-2.5-hardening.md`
- 对照依据：`reviews/2026-04-10-phase-2-cto.md`
- 参考延后项：`reviews/phase-1-deferred.md`、`reviews/phase-0-deferred.md`

**一句话结论**：
这份 `Phase 2.5` playbook 方向基本对，但还没有完全抓住“Flow Hardening”最该先修的东西。  
它已经覆盖了 DB 唯一性、`server-only`、音频初始化、时钟源、角色分离这些重要项；但从 CTO 视角看，**它现在更像“平台硬化清单”，还不是一份完全对准 Phase 2 核心风险的“交互事务硬化计划”**。

---

## 发现

### [P0] 这份 Flow Hardening sprint 漏掉了上一轮 CTO review 里最优先的两项流转真相

位置：
- `playbook/phase-2.5-hardening.md:15-18`
- `reviews/2026-04-10-phase-2-cto.md:245-262`

问题：
上一轮 CTO review 已经把 Phase 2.5 的第一优先级写得很明确：

1. 爱心收藏状态机收口
2. 明确“无草稿是否允许收藏”
3. 红心本地缓存与真实后端状态对齐
4. 草稿 TTL 规则文案和服务端计算一致

但当前 playbook 只保留了其中两类“偏底座”的事项：

- DB 唯一索引
- TTL 文案对齐

真正最危险的交互事务问题反而不在计划里：

- 爱心现在还是多副作用按钮，没有状态机 step
- “没演奏能不能收藏”仍然没有被产品规则钉死
- 红心本地缓存的真相对齐没有单独步骤

这会导致一个很危险的结果：  
你们名义上在做 `Flow Hardening`，但真正最脆弱的 flow 没有进入执行清单。

建议：
- 在 `S0` 前面插一个 `S0a`，或直接把 `S0` 重写成“交互事务真相收口”
- 至少显式包含：
  - 爱心状态机
  - 无草稿时的收藏规则
  - 本地红心缓存与后端状态对齐
  - token log 删除

---

### [P0] `pending_scores` 的唯一索引迁移缺少“脏数据清理 / 回填”步骤，线上有旧脏数据时会直接卡死

位置：
- `playbook/phase-2.5-hardening.md:25-33`

问题：
当前 `S0` 直接写：

`CREATE UNIQUE INDEX ... ON pending_scores (user_id, track_id) WHERE status = 'draft'`

但 playbook 没有交代一个关键前提：  
**如果数据库里已经存在重复的 `draft` 行，这个索引会直接创建失败。**

考虑到你们 Phase 2 之前确实没有 DB 层唯一约束，而且曾经有：

- `/me` 自动上传
- 爱心上传
- 多标签页并发

那么“已有重复 draft”不是理论风险，而是现实可能。

建议：
- 在 migration 前增加数据清理策略
- 最低限度要写清楚：
  - 保留最新一条还是最早一条
  - 其余重复 draft 如何批量标成 `expired`
  - 索引创建失败时如何回滚

更稳的写法是：
1. 先清理重复 draft
2. 再创建部分唯一索引

### [P1] `S0` 说要“抽成共享 TTL 常量”，但范围里并没有包含真正的后端真理源文件

位置：
- `playbook/phase-2.5-hardening.md:24-34`

问题：
当前 `S0` 的范围只有：

- `HomeJam.tsx`
- `draft-store.ts`
- `supabase.ts`
- `operator-wallet.ts`
- migration

但真正决定 TTL 规则的后端文件其实是：

- `app/api/score/save/route.ts`
- 以及必要时的 `src/types/jam.ts`

如果不把这些文件收进范围，你们很容易得到一个“前端抽了常量、后端还是自己的 TTL_MS”的伪统一。  
这会让 `S0` 的标题叫“TTL 文案对齐”，结果代码层仍然有两套真理源。

建议：
- 把 `app/api/score/save/route.ts` 明确纳入 `S0`
- 如果真要共享常量，最好新建一个明确的 shared module，而不是只改前端文案

---

### [P1] `S2` 仍然保留 `performance.now()` fallback，会把“双时钟语义”继续留在系统里

位置：
- `playbook/phase-2.5-hardening.md:67-73`

问题：
`S2` 的主方向是对的：把录制事件时间迁到音频时钟。  
但 step 3 又写：

“没有背景曲播放时（纯键盘演奏），回退到 `performance.now()`”

这会留下一个新的长期问题：

- 一部分事件来自音频时钟
- 一部分事件来自页面时钟

如果未来这些事件都可能进入同一个 `score_data` 模型，你们还是会带着“双语义事件时间”进入 Phase 3。

而且按当前产品规则，“没播放背景曲时按键”本来就不应该形成正式草稿；  
那这个 fallback 很可能是在给一个不需要存在的路径继续加复杂度。

建议：
- 要么明确：无背景曲时不录制，因此不需要 fallback
- 要么明确：即使无背景曲也录制，但仍然要定义可映射到统一回放真相的时间源

我的建议是前者：  
**Flow Hardening 的目标是消除双语义，不是保留双语义。**

### [P2] `server-only` 被写进执行步骤，但没有和技术栈白名单 / 环境变量校验一起收口

位置：
- `playbook/phase-2.5-hardening.md:26-34`
- `docs/STACK.md:3`
- `reviews/phase-1-deferred.md:12`
- `reviews/phase-0-deferred.md:13`

问题：
`S0` 计划里写了：

- 给 `supabase.ts` / `operator-wallet.ts` 加 `import 'server-only'`
- 并且显式写了 `npm install server-only --legacy-peer-deps`

但这里有两个缺口：

1. `server-only` 不在当前 `STACK.md` 白名单里
2. 之前 deferred 里一直是“`server-only` + 环境变量启动校验”成对出现，现在只做前者，不做后者

这不是说 `server-only` 不该做，而是说这一步目前写得不够闭环。  
从项目协议看，新增依赖至少要么更新栈文档，要么在 playbook 里标明“需用户批准”。

建议：
- 把 `server-only` 的栈登记一并写进本 step
- 顺手把环境变量启动校验并入 `S0`，否则这个 step 只修了一半边界

---

### [P2] `S3` 把“合约角色分离”和“统一 build 验证”绑在一个 step，风险性质不同，完成标准也会变模糊

位置：
- `playbook/phase-2.5-hardening.md:82-106`

问题：
`S3` 当前同时包含两件完全不同的事：

- 安全边界：`admin / minter` 分离
- 工具链 / 发布纪律：`verify.sh` 加 `npm run build`

这两个任务：

- 风险不同
- 修改范围不同
- 验证方式不同
- 回滚方式也不同

绑在一起的结果通常是：
- 一个做完了，另一个没做完
- 但 step 名义上又像“都做完了”

建议：
- 拆成 `S3a 合约角色分离`
- 和 `S3b 统一 build / verify`

这样对 Claude/Codex/用户都更清楚，也更符合“一次只做一件事”的项目纪律。

---

## 整体判断

这份 `Phase 2.5` playbook 有明显的优点：

- 比较短，节奏克制
- 没有趁 hardening sprint 偷偷塞新功能
- 抓住了三个真正重要的底座问题：唯一性、音频初始化、时钟源
- 对 Phase 3 的前置门槛意识是对的

但它现在还有一个明显的问题：

**它更像“平台/工程硬化”，还不够像“Flow Hardening”。**

也就是说：

- 它抓住了技术债
- 但还没完全抓住产品语义债

而 Phase 2 留下的最危险问题，恰恰是“产品语义债”。

---

## 我会怎么改这份 playbook

如果我来重排，我会改成这样：

### S0a：爱心收藏 flow 真相收口
- 定义爱心状态机
- 定义“无草稿能否收藏”
- 红心缓存和后端真相对齐
- 删除浏览器 token log

### S0b：DB + server-only + TTL 真理源统一
- 清理重复 draft
- 创建部分唯一索引
- `server-only`
- 环境变量校验
- TTL 常量统一到真正共享源

### S1：AudioContext 延迟加载
- 只处理首页合奏引擎初始化问题

### S2：录制时钟源统一
- 明确是否允许无背景曲录制
- 事件时间只保留一套语义

### S3a：admin/minter 分离
- 只做合约权限和部署脚本

### S3b：统一 build / verify
- `npm run build`
- 生产构建纳入 gate

---

## 最终判断

如果现在就照这版 playbook 开工，**大概率能修掉一部分重要技术债**。  
但我不认为它已经足够准确地命中了“Phase 2 为什么还不能安心进 Phase 3”的核心。

我这轮最核心的建议只有一句：

**先把这份 `Phase 2.5` playbook 从“平台硬化清单”修成“交互事务 + 平台底座的双重硬化计划”，再开工。**

如果你把这份文档给 Claude，我建议直接让它做两件事：

1. 按这份 review 把 `phase-2.5-hardening.md` 改成 v2
2. 单独补一节“爱心收藏状态机”，把 allowed / disallowed flow 写死
