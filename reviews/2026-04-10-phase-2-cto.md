# CTO Review 2026-04-10 — Phase 2

**审查定位**：这不是普通的“功能有没有跑通”检查，而是站在 CTO / 产品技术负责人的视角，判断 Phase 2 完成后，这个项目是否已经从“可玩的 MVP”升级成“可以继续资产化、公开化、复杂化”的产品底座。

**审查范围**：
- Phase 2 当前代码实现：首页合奏、草稿、爱心收藏、个人页、草稿 API、现有 mint 队列
- 当前架构意图：`docs/ARCHITECTURE.md`
- 当前执行状态：`STATUS.md` / `TASKS.md`
- 跨 Phase 底座：合约权限、服务端边界、验证脚本、健康检查

**一句话结论**：
Phase 2 是项目第一次真正像“作品”，而不只是“能跑的 Web3 demo”。但从 CTO gate 的标准看，**它还不是一个可以无脑进入 Phase 3 的状态**。现在最需要补的不是更多玩法，而是把“创作 → 收藏 → 草稿 → 资产化”这条链路里的真实语义和真实状态机钉死。

---

## 一、我认可的部分

### 1. 首页即乐器，这个方向是对的

我认可这次对 Patatap 哲学的吸收。  
从产品体验上看，“首页就是乐器”明显比单独 `/jam` 路由更强：

- 零跳转，进入即玩
- 用户先感受作品，再理解功能
- 音效、视觉、岛屿、播放不再是分裂页面

这是 Phase 2 最重要的升级，也是项目第一次真正具备“作品入口页”的气质。

### 2. 你们已经做出了真正的交互闭环

当前 Phase 2 已经不只是页面上多了键盘和动画，而是形成了完整链路：

- 首页按键触发音效和视觉
- 点击岛屿播放背景曲
- 播放生命周期绑定录制
- 草稿落到本地并带 TTL
- 登录后上传草稿
- 爱心触发收藏 / 铸造
- 个人页展示收藏和创作记录

这说明项目已经从“听音乐 + 点一下 mint”进化成“能参与创作的音乐作品”。

### 3. 这轮开发纪律比之前更像真正的产品研发

我也认可这轮几个工程判断：

- Phase 2 没把 Arweave 硬塞进主链路
- 先做 spike，再做首页融合
- 草稿、音效、收藏、播放条之间有明确的分工
- `verify.sh` 这次能稳定跑过，说明交付纪律比前几轮成熟

这不是小事。产品越走到后面，越需要团队开始相信自己的 release gate。

## 二、发现

### [P0] 爱心收藏在真正成功前就被写成“已成功”，失败时不会回滚

位置：
- `src/hooks/useFavorite.ts:29-31`
- `src/components/archipelago/Archipelago.tsx:43-46`
- `src/components/FavoriteButton.tsx:23-33`

问题：
当前 `useFavorite` 在真正拿到任何后端成功结果之前，就先：

1. `setStatus('success')`
2. 调 `onMinted(tokenId)`

而 `Archipelago` 收到 `onMinted` 后，会立刻把 tokenId 写进本地状态和缓存。  
这意味着：

- 登录失败，红心也可能已经亮了
- `/api/mint/material` 写队列失败，红心也可能已经亮了
- 草稿上传失败、收藏流程半路出错，本地仍会把它当成“已收藏”

这不是纯 UI 细节，而是会污染用户心智和本地状态的“假成功”。

建议：
- 把当前实现改成真正的状态机：`idle / auth / uploading / minting / success / error`
- 只有在后端确认“收藏/铸造请求已成功进入系统”之后，才写红心和缓存
- 如果坚持乐观 UI，也必须配套明确的失败回滚

### [P1] 代码实际上允许“没演奏也能收藏”，和 Phase 2 的主叙事没有完全对齐

位置：
- `src/hooks/useFavorite.ts:37-69`
- `docs/ARCHITECTURE.md` 当前 Phase 2 路径描述

问题：
现在爱心点击的真实顺序是：

1. 先调 `/api/mint/material`
2. 再看本地有没有草稿
3. 有草稿就顺手上传，没有就算了

也就是说，**没有演奏、没有草稿，也能爱心收藏素材 NFT**。  
如果这是产品有意为之，那没问题；但如果 Phase 2 想强调的是“创作后再收藏”，那现在代码并没有强制这件事。

建议：
- 明确写死一个产品规则：
  - 规则 A：允许纯收藏素材，创作只是附加价值
  - 规则 B：必须先创作，才能收藏
- 规则没钉死之前，后面做 Phase 3 的 ScoreNFT 语义会一直摇摆

### [P1] `pending_scores` 的唯一性规则还没有被数据库真正兜住

位置：
- `app/api/score/save/route.ts:119-162`
- `supabase/migrations/005_pending_scores.sql:4-18`

问题：
当前 API 逻辑假设“同一 user + track 只会有一条 `status='draft'` 的有效草稿”：

1. 先把旧草稿标 `expired`
2. 再插入新草稿
3. 还写了 `23505` 的唯一键冲突兜底

但数据库迁移里并没有相应的唯一约束。  
所以并发情况下，仍然可能留下多条有效草稿。  
只要 `/me` 自动上传、本地上传、爱心上传在多个标签页里同时发生，这个假设就会破。

建议：
- 在 DB 层补真正的唯一约束或部分唯一索引
- 让“有效草稿只能有一条”成为数据层事实，而不是 API 幻觉

### [P1] 录制时间仍然基于页面时钟，不是音频时钟，这会在 Phase 3 放大成结构性债务

位置：
- `src/hooks/useRecorder.ts:37-40`
- `src/hooks/useRecorder.ts:91`
- `src/hooks/useRecorder.ts:128`

问题：
当前事件时间基准是 `performance.now()`，不是背景曲播放所使用的音频时钟。  
在本地 demo 阶段这可能“看起来够用”，但只要 Phase 3 要做：

- `mint_events.score_data`
- 公开回放页
- Score Decoder
- 链上资产化回放一致性

你就会发现“页面时钟”和“音频时钟”不是一回事。  
掉帧、tab 切换、浏览器调度、不同 context 的启动时间，都会慢慢放大这笔债。

建议：
- 尽快统一到音频时钟体系
- 如果背景曲和键盘音效暂时不共用一个 transport，也至少要定义一个“可映射到回放真相”的统一时间源

### [P1] 首页合奏引擎仍然在页面加载阶段创建 `AudioContext` 并预解码整套音效

位置：
- `src/hooks/useJam.ts:39-49`
- `docs/CONVENTIONS.md` 关于 `AudioContext` 的规则

问题：
当前 `useJam` 在 `useEffect` 中直接 `new AudioContext()` 来做解码。  
这有两个问题：

1. 和你们自己“页面加载时禁止创建 AudioContext”的规则冲突
2. 让用户即使只是想先看首页，也要立刻承担 26 个音效的加载和解码成本

作为作品入口页，这会影响：

- 首屏响应
- CPU / 内存占用
- 浏览器兼容性和设备差异

建议：
- 把真正的解码/初始化尽量延后到首次用户手势之后
- 或至少把音效资源分层：先能浏览，进入交互后再 fully ready

### [P2] Phase 2 已完成，但浏览器里仍然保留调试级 token 输出

位置：
- `src/components/auth/LoginButton.tsx:13-18`

问题：
登录后仍会把 `evm_address` 和 `privy_token` 输出到 console。  
这在 demo、录屏、共享电脑、远程协作场景里都不专业，也不应该再出现在“作品级体验”的阶段。

建议：
- 直接删掉
- 如果确实还需要调试入口，做一个显式的 dev-only 开关，而不是默认暴露给所有浏览器会话

## 三、第一轮没完全展开的 CTO 视角

### 1. 现在最大的风险，已经不是“功能没做完”，而是“交互真相和系统真相可能不一致”

Phase 2 最危险的问题不在美术、不在音效、不在页面跳转。  
真正的风险是：

- 用户看到红心了，系统不一定真的记住了
- 用户创作过了，系统不一定按真实创作时间算 TTL
- 用户有草稿了，个人页不一定能完整表示这份草稿的真实状态

顶级产品里，作品感和可信度是同时成立的。  
只做氛围，不做真相对齐，后面一定会反噬。

### 2. Phase 2 已经让“创作数据”变成第一等资产

Phase 1 的核心资产是 mint 结果。  
Phase 2 开始，真正珍贵的东西变成了：

- 用户什么时候创作
- 创作了哪些事件
- 这段创作和哪首曲目绑定
- 这份草稿何时过期、何时上链、是否进入个人收藏

也就是说，系统的第一等资产开始从“交易结果”扩展到“创作过程数据”。  
这意味着下一步不能再只用 Phase 1 的队列思维，而要开始有“作品数据生命周期”的视角。

### 3. 品牌体验第一次成立了，但审美系统还没有真正封装成设计系统

我认可首页的情绪和方向已经建立起来了。  
但从更高标准看，Phase 2 现在更像“一个好用的交互实验”，还不是“一个有完整视觉语言的作品系统”。

具体表现为：

- 视觉反馈更偏功能性，不够“仪式化”
- 首页字体和整体排版还没有形成品牌识别
- 移动端现在是“可浏览，不可玩”的让步，而不是一个被明确设计过的 mobile 叙事

这不是 bug，但它会决定你 Phase 3 做出来的是“厉害功能”，还是“难忘作品”。

### 4. 跨 Phase 底座债务仍然还在

虽然这轮主要审 Phase 2，但有两笔老债还没消失：

- 合约 `admin / minter` 分离仍未彻底落地
- `supabase.ts` / `operator-wallet.ts` 还停留在“靠注释说不要前端 import”，没有 `server-only`

如果你们下一步做 Phase 3 的链上资产化，这两笔债会重新浮上来，而且比现在更贵。

## 四、我对下一步开发计划的建议

我不建议直接把下一步定义成“进入 Phase 3”。  
我建议先插入一个非常短的：

**Phase 2.5 — Flow Hardening Sprint**

### 第一组：交互事务真相（必须先做）

1. 爱心收藏状态机收口
2. 明确“无草稿是否允许收藏”
3. 红心本地缓存与真实后端状态对齐
4. 草稿 TTL 规则文案和服务端计算完全一致

### 第二组：创作数据可信度（紧随其后）

1. `pending_scores` 的 DB 唯一约束
2. 录制时间源从页面时钟迁到音频时钟或可映射时钟
3. `/me` 的草稿语义明确区分：本地草稿 vs 已上传草稿

### 第三组：跨 Phase 清债（进入 Phase 3 前做）

1. 删除浏览器 token log
2. `server-only` 加到服务端能力模块
3. `admin / minter` 分离方案重新核对
4. 补统一 `typecheck` / `build` 脚本入口，不只靠 `verify.sh`

## 五、我会怎么定义真正的 Phase 3

不要把 Phase 3 定义成“更多链上能力”。  
我会把它定义成：

**Phase 3 — 可信作品资产化**

顺序建议是：

1. 先做数据层：
   - `score_data` 结构冻结
   - 公开回放数据模型
   - Score Decoder 的输入协议

2. 再做内容层：
   - 封面分配
   - ShareCard
   - 公开 `/score/[tokenId]`

3. 最后做链上层：
   - ScoreNFT
   - MintOrchestrator
   - ERC-6551
   - 链上补偿和幂等

如果顺序反过来，你们很容易先把链上东西做复杂，再回来重修数据真相。

## 六、最终判断

我对 Phase 2 的评价，比对 Phase 1 结束时更高。  
因为你们已经证明：这个项目不只是“可以 mint 的音乐页”，而是真正开始长出“创作型作品”的形状。

但我也会更严格。  
因为一旦作品开始承载用户创作数据，标准就必须升级：

- 不只是“能不能玩”
- 而是“记录的是不是可信”
- 不只是“能不能收藏”
- 而是“收藏和创作之间的语义是不是一致”

所以我这轮最核心的建议只有一句：

**下一步不要直接定义成“进入 Phase 3”，而要先把 Phase 2 从“可玩的作品”升级成“可信的作品系统”，再进入真正的资产化阶段。**
